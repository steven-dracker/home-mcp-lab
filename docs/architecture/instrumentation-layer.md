# Instrumentation Layer — Home MCP Compliance Lab

**Status:** Draft  
**Date:** 2026-03-31  

---

## Summary

This document defines the instrumentation layer for the Home MCP Compliance Lab — the mechanism by which real system behavior is translated into audit events that enter the platform ingestion workflow. The instrumentation layer sits between the operating system (MCP servers, agents) and the platform boundary. It is the missing piece that closes the gap between the platform's audit event schema and observable system behavior.

The instrumentation layer is not yet implemented. This document defines what it must do.

---

## Purpose

The platform owns audit observability. The audit event ingestion workflow (`docs/workflows/audit-event-ingestion.md`) defines what happens after an event is emitted. But nothing currently generates events. The instrumentation layer is the answer to:

> Who emits audit events, when, and how?

Without a defined instrumentation layer:
- Audit events cannot flow
- The ingestion workflow has no input
- Control patterns cannot be validated
- Visibility gaps cannot be closed

This document defines the instrumentation layer as a platform-owned behavior specification.

---

## Instrumentation Model Overview

The instrumentation layer operates at two distinct boundaries:

**1. MCP Server Boundary**
Events are generated at the point where a tool call enters or exits an MCP server. This is the primary source of `tool.invocation` events. Instrumentation at this boundary captures: what tool was called, by what agent, with what non-sensitive context, and what the outcome was.

**2. Agent Execution Layer**
Events are generated by the agent (or on behalf of the agent) to establish session context. This is the source of session lifecycle events. Instrumentation at this layer establishes the `correlation_id` that links all tool call events within a session.

Both boundaries are required for a complete audit trace. MCP server instrumentation without agent-layer session events produces uncorrelated tool call records. Agent-layer session events without MCP server instrumentation produce session boundaries with no content.

The instrumentation layer must be platform-owned. Individual MCP servers must not implement proprietary instrumentation — they must conform to platform instrumentation expectations.

---

## Event Generation Points

The following points in system execution must generate audit events:

| Event Generation Point | Event Type | Emitter |
|---|---|---|
| Agent initiates MCP session | `session.start` | Agent / instrumentation layer |
| MCP tool call begins | `tool.invocation` (start) | MCP server / instrumentation layer |
| MCP tool call completes (success) | `tool.invocation` (success) | MCP server / instrumentation layer |
| MCP tool call completes (failure) | `tool.invocation` (failure) | MCP server / instrumentation layer |
| Service or agent requests a secret | `secret.retrieval` (attempt) | Service / instrumentation layer |
| Secret retrieval completes (success or failure) | `secret.retrieval` (outcome) | Service / instrumentation layer |
| Agent closes or exits MCP session | `session.end` | Agent / instrumentation layer |

Events must be generated as close to the action as possible — ideally at the exact moment the action begins and again when it concludes. This enables duration calculation and distinguishes actions that start from actions that complete.

## MCP Server Instrumentation

### Responsibilities

MCP servers (or the instrumentation layer wrapping them) must emit audit events for:

- Every tool call invocation — regardless of outcome
- Every interaction with an external system triggered by a tool call (e.g., GitHub API call, database query) where that interaction is observable at the server boundary

### Required Behavior

- Events must be emitted for both the start and completion of each tool call
- Failure events must be emitted when a tool call raises an error, times out, or returns a non-success status
- Events must conform to `schemas/audit-event.schema.json`
- Events must include the `correlation_id` established at session start
- Events must include non-sensitive argument context in `metadata` — sufficient to identify what the tool was doing without exposing sensitive values

### Prohibited Behavior

- MCP servers must not include raw tool arguments in events if those arguments may contain sensitive data (file contents, API responses, secret values)
- MCP servers must not define their own event schemas or emit events in non-conforming formats
- MCP servers must not suppress or skip event emission for specific tool calls
- MCP servers must not allow event emission failures to block or fail the tool call itself

### Instrumentation Approach Options

MCP servers may achieve conforming instrumentation through one of the following approaches (decision to be made via ADR):

1. **Native emission** — the MCP server is modified or configured to emit platform-conforming events directly
2. **Platform proxy** — a platform-owned proxy intercepts all tool calls and emits events without modifying the MCP server
3. **Agent-side emission** — the agent (Claude Code) emits events for each tool call it initiates, without involving the MCP server

The choice of approach is an architectural decision outside the scope of this document.

## Agent-Level Instrumentation

### Responsibilities

Agents are responsible for:

- Emitting `session.start` events when an MCP session is initiated
- Emitting `session.end` events when a session concludes
- Generating and propagating the `correlation_id` for the duration of the session
- Including sufficient session context in lifecycle events to make the session attributable and traceable

### Required Behavior

- `session.start` must be emitted before any tool calls are made within the session
- `session.start` must establish the `correlation_id` that all subsequent events in the session will use
- `session.end` must be emitted when the session concludes, whether successfully or due to error
- If the agent terminates abnormally without emitting `session.end`, the platform must be able to detect this as an incomplete session
- Agent identity (`agent_id`) must be declared in `session.start` and must match the `agent_id` field in all tool call events within the session

### Context Propagation

The agent is the authority for session context. It must propagate the following to all operations within a session:

- `correlation_id` — the session identifier linking all events
- `agent_id` — the identity of the agent or process initiating the session
- `project_id` — the project context in which the session is operating
- `mcp_server` — the MCP server being connected to

These values must be consistent across all events within a session. Inconsistency is a validation failure.

## Event Emission Requirements

All emitted events must:

- Conform to `schemas/audit-event.schema.json` in structure and field types
- Include all required top-level fields
- Use a recognized `event_type` value
- Set `platform` to `"home-mcp-lab"`
- Include a `correlation_id` that matches the session in progress
- Be emitted in real-time or near-real-time — not batched across sessions or delayed beyond the completion of the action
- Never include secret values, credentials, tokens, or PII in any field, including `metadata`

Events must not:
- Omit required fields
- Use placeholder or null values for required identity fields (`project_id`, `agent_id`, `mcp_server`)
- Be emitted retroactively for past actions (reconstructed events are not audit events)

---

## Event Timing and Lifecycle

For operations that have a start and end (tool calls, secret retrievals), two events must be emitted:

**Start event:**
- Emitted immediately before the operation begins
- `status` field: not applicable at this point — use a defined pending or in-progress convention if needed, or omit until schema supports it
- Establishes a record that the operation was initiated

**End event:**
- Emitted immediately after the operation completes (success or failure)
- `status` field: `success` or `failure`
- `metadata` may include execution duration (calculated from start event timestamp)

The pairing of start and end events enables:
- Duration measurement
- Detection of operations that start but never complete (hangs, crashes)
- Clear separation of invocation from outcome

For instantaneous operations (no meaningful duration), a single event at completion is acceptable.

## Correlation and Context Propagation

### Correlation ID Lifecycle

1. The agent generates a unique `correlation_id` at session start
2. The `correlation_id` is included in the `session.start` event
3. The `correlation_id` is passed to every tool call made within the session
4. Every tool call event includes the same `correlation_id`
5. The `session.end` event includes the `correlation_id`

The result is a complete, linked chain of events from session start to session end, queryable by `correlation_id`.

### Multi-Session Workflows

Some workflows span multiple sessions (e.g., a multi-step task over several days). For these:
- Each session has its own `correlation_id`
- A higher-level workflow ID (e.g., a CC-HMCP prompt ID) may be included in `metadata` to link related sessions
- The platform must not assume all related work occurs in a single session

### Orphaned Correlation IDs

If a session emits a `session.start` event but never emits a `session.end` event within a defined timeout window, the platform treats this as an incomplete session. Incomplete sessions are a visibility signal — they indicate abnormal termination and must surface in gap detection.

---

## Failure and Fallback Behavior

### Non-Blocking Requirement

Event emission must never block or fail the primary operation. If the instrumentation layer fails to emit an event:
- The tool call, secret retrieval, or session operation proceeds normally
- The emission failure is logged locally at the instrumentation layer
- The platform's gap detection layer may identify the missing event later

Instrumentation failures must not cascade into operational failures.

### Emission Retry

If an event cannot be submitted to the ingestion boundary (e.g., network failure):
- The instrumentation layer must retry submission
- Retry must use a bounded backoff strategy
- Events that cannot be delivered after exhausting retries are logged as lost events
- Lost events represent an audit gap and must be surfaced in platform observability

### Partial Instrumentation

In cases where only some event generation points are instrumented (e.g., tool completion but not tool start), the platform must treat the partial record as better than nothing while marking the session as incompletely instrumented. Partial instrumentation is preferable to no instrumentation, but must not be presented as complete compliance.

## Security Considerations

- **No secrets in payloads** — the instrumentation layer must inspect all outbound events and reject or redact any field that contains probable secret patterns before submission to the ingestion boundary
- **Minimal metadata exposure** — `metadata` fields must contain only what is necessary to identify the action; raw tool arguments, API responses, and file contents must not be included unless explicitly confirmed to be non-sensitive
- **Audit trail integrity** — the instrumentation layer must not modify events after emission; once an event is submitted, it is immutable
- **Instrumentation layer identity** — the instrumentation layer must authenticate to the ingestion boundary to prevent event injection from unauthorized sources (mechanism to be defined via ADR)
- **No instrumentation bypass** — it must not be possible for an MCP server or agent to operate outside the instrumentation layer without that absence being detectable

## Known Gaps and Limitations

The instrumentation layer described in this document does not yet exist. The following visibility gaps directly prevent implementation:

**VG-HMCP-000002 — Tool Invocation Visibility**
No mechanism exists to intercept or wrap MCP tool calls for event emission. The instrumentation approach (proxy, native, agent-side) has not been decided. Until an ADR selects an approach, this gap cannot be closed.

**VG-HMCP-000003 — Secret Retrieval Path Visibility**
The current secret retrieval path for the GitHub MCP server is undocumented. The instrumentation layer cannot emit `secret.retrieval` events for a retrieval path it cannot observe.

**VG-HMCP-000004 — Session Lifecycle Visibility**
No `session.start` or `session.end` event types are defined in `schemas/audit-event.schema.json`. Without these types, agent-level instrumentation cannot produce conforming events. The schema must be extended before session lifecycle instrumentation can be implemented.

**Overall:** The instrumentation layer is fully defined at the behavioral level but has zero implementation. This document is the specification against which future implementation work will be validated.
