# Workflow and Operating Rules — [PROJECT NAME]

This document defines the operating conventions for Claude Code and ChatGPT sessions in this repository.

These rules are durable. They do not change based on execution state. For current state, see the latest handoff in `handoffs/`.

---

## Roles

| Role | Party | Responsibilities |
|---|---|---|
| Architect / Scope Enforcer | ChatGPT | Maintain platform discipline, generate scoped prompts, review execution summaries, generate PR text, guide the system forward |
| Executor / Repo Modifier | Claude Code | Execute prompts exactly as scoped, commit changes, generate handoffs, report state accurately |
| Operator / Controller | User | Run prompts in Claude Code, return summaries to ChatGPT, approve and merge PRs, control session boundaries |

---

## What Claude Code Is Expected To Do

- Execute only what is defined in the prompt — no scope expansion
- Follow branch name and commit message exactly as specified
- Never commit directly to main
- Generate a handoff at session boundaries
- Report execution summaries in the required format
- Document assumptions when ambiguity is unavoidable
- Stop and report if a task cannot be completed as specified

## What ChatGPT Is Expected To Do

- Act as architect and scope enforcer — not implementer
- Generate tightly scoped, copy-safe prompts
- Review execution summaries before generating PR text
- Enforce branch/PR discipline
- Maintain consistency across sessions
- Flag scope drift or ambiguity before prompting Claude Code

## What The Operator Is Expected To Do

- Run Claude Code prompts without paraphrasing or modifying them
- Return execution summaries to ChatGPT verbatim
- Approve and merge PRs after review
- Initiate session boundaries by requesting handoffs
- Control the pace of work — neither Claude Code nor ChatGPT initiates the next task unilaterally

---

## Standard Execution Loop

```
Prompt → Execute → Summary → Review → PR → Merge → Handoff → Stop
```

1. ChatGPT generates a scoped prompt
2. Operator runs the prompt in Claude Code
3. Claude Code returns a structured execution summary
4. Operator returns the summary to ChatGPT
5. ChatGPT reviews the summary and generates PR title + description
6. Operator opens and merges the PR
7. Claude Code generates a handoff if a session boundary is reached
8. All parties stop; operator controls when to resume

---

## Branch and PR Discipline

- Feature branches only — never commit directly to main
- Branch names follow: `feature/<short-task-name>`
- Commit messages follow: `<type>: <summary>` (feat, fix, docs, refactor, chore)
- PR process is mandatory: branch → push → PR → review → merge → delete branch → sync main
- PR text is generated by ChatGPT after reviewing the execution summary — not before

---

## Session Boundary Discipline

Create a handoff when:
- A logical unit of work is complete (PR merged)
- Context is running low (approximately 30% remaining)
- A natural pause point is reached (end of phase, end of session)

Do not continue to the next task across a session boundary without a handoff.

Handoffs are the persistent state layer. Primers carry durable rules. The two must not be confused.

---

## Resume Pattern

1. Open a new ChatGPT session
2. Paste `chatgpt-primer.md` (adapted from `chatgpt-primer.template.md`)
3. Paste the latest handoff from `handoffs/`
4. ChatGPT restores context from the handoff
5. Continue from the next recommended task

Do not rely on ChatGPT or Claude Code memory across sessions. The handoff is the source of truth.

---

## What Primers Are For

Primers carry durable behavioral guidance:
- role model
- workflow model
- branch/PR discipline
- resume pattern
- stable project context

Primers do not track live state. Do not update the primer to reflect current execution state — that belongs in the handoff.

---

## What Handoffs Are For

Handoffs carry dynamic execution state:
- last completed task
- active branch and PR status
- open visibility gaps or risks
- next recommended task
- recent key artifacts

The most recent handoff is always authoritative for dynamic state. If primer and handoff conflict, the handoff wins.

---

## Prompt Discipline

All Claude Code prompts must:
- begin with a traceable prompt ID (e.g., `CC-PROJECT-000001`)
- specify branch name and commit message exactly
- be tightly scoped — no hidden scope expansion
- be copy-safe (no nested triple backticks, no broken markdown)

Prompts that do not meet this standard should be regenerated before execution.

---

## Execution Summary Format

Claude Code must return a structured execution summary including:
- Prompt ID executed
- Summary of what was done
- Files created or modified
- Notes / assumptions
- Validation confirmation
- Branch name and commit hash
